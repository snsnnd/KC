# 系统架构

## 运行组件

- Nginx 提供 `public/` 静态页面、HTTPS、`/api/` 反向代理和 `/uploads/` 文件访问。
- Node.js + Express 提供内容、申请、成员权限、邮件、库存、资金和审批 API。
- `/portal.html` 是管理员与成员的统一身份入口；管理控制台通过查询参数划分运营、人员、资源资金三个工作区。
- systemd 以 `www-data` 用户运行单个 Node.js 实例。
- `DATA_DIR` 保存 JSON 运行数据，`UPLOAD_DIR` 保存媒体文件。

## 持久化边界

当前持久化是纯文件方案，不是数据库。普通 JSON 通过临时文件写入后原子重命名；SMTP 授权码和受保护资源密码使用 `SESSION_SECRET` 派生的密钥进行 AES-256-GCM 加密。

库存、资金和申请审批共享进程内串行锁，保证单实例中不会重复审批或超额扣减。禁止在没有迁移到数据库事务的情况下启动多个 API 实例。

成员可见模块由服务端权限计算决定。资源支持主链接、多个附加链接和最多三级的合集树；子资源必须同时满足所有祖先合集和自身的权限标识，详情接口只解密当前节点的提取码，不会批量下发后代凭据。公开 API 会递归清空受保护节点及其后代的真实链接。没有材料或资金申请权限时，对应库存和账户数据不会下发。前端隐藏仅用于体验，所有详情读取和提交操作仍由 API 再次鉴权。

管理员采用双重范围控制：`panelPermissions` 决定可管理模块，`departmentIds` 决定可见部门数据。`owner` 隐式拥有全部模块和全部部门；非主管理员缺少授权时服务端返回 `403`，跨部门对象使用 `404` 防止枚举，前端不会渲染对应入口或面板。

`notifications` 是所有管理员的隐式基础面板。非主管理员的通知收件人只能来自其 `departmentIds` 范围内的有效成员和未转成员申请人；主管理员才可加入管理组、默认邮箱组或自定义外部邮箱。成员、申请人和外部收件地址合并到同一集合去重后发送。

成员问询保存在 `member-messages.json`。成员只能读取、创建和回复自己的线程；成员回复已关闭线程时会在串行锁内将其重新打开并再次通知负责管理员。管理员通过隐式通知权限读取负责部门内的线程，回复操作在串行锁内重新验证管理员部门范围。双方消息保存在同一线程，管理员回复还会尝试发送到成员档案中的当前邮箱。

新账号以成员身份为唯一凭据，登录名由部门 `accountPrefix` 与学号组合。创建账号时签发 10 位随机一次性激活码，`member-activation-codes.json` 只保存基于 `SESSION_SECRET` 的 HMAC 哈希、有效期和使用状态。成员在独立的 `/activate.html` 页面使用激活码设置自己的初始密码后，`mustChangePassword` 才会清除；激活接口不会签发会话 Cookie，成员需返回登录页手动登录。激活前不能登录、读取资源或获得后台权限。`admins.json` 中的新管理员记录通过 `memberId` 关联成员，只保存后台角色和范围；认证密码来自成员记录。没有管理员关联记录的成员不能进入管理端。历史管理员记录没有 `memberId` 时继续使用原有密码哈希。

申请通知配置保存具体管理员 ID，并在发送时解析其当前邮箱。普通通知邮箱组与申请负责人是两个独立概念，避免仅凭邮箱地址无法确认责任人。

加入申请和资源使用申请共用 `email-approval-tokens.json`，通过 `kind` 区分业务对象。加入申请 Token 同样绑定管理员、账号版本、收件邮箱、申请和通过/拒绝动作；后台或邮件任一通道完成审批后，同申请的其他 Token 全部失效。加入申请通过后仍需单独执行“转为成员”才能签发统一账号和激活码。

资源使用邮件审批 Token 使用 32 字节随机值，数据目录仅保存基于 `SESSION_SECRET` 的 HMAC-SHA256 哈希。Token 绑定申请、审批管理员、账号版本、收件邮箱和批准/拒绝动作，默认 24 小时过期；预览请求不改变状态，确认请求与后台审批共用资源串行锁，成功后使同申请全部 Token 失效。管理员停用、删除、改密、邮箱变化、角色/模块/部门范围变化都会在同一锁内使旧 Token 失效。邮件链接把 Token 放在 URL Fragment 中，避免进入代理访问日志和 Referer。

审批结果邮件只发送到成员档案中的当前邮箱，不回退到申请快照中的旧邮箱。批准和拒绝模板保存在加密邮件配置中，后台审批与邮件链接审批使用相同的单次占位符渲染逻辑。

## 数据模型

- `content.json`：站点设置、项目、部门，以及带多个链接和子节点的资源树。
- `applications.json`：加入申请，含姓名、学号、班级和联系方式。
- `members.json`：统一登录账号、基础资料、密码状态和成员权限集合。
- `admins.json`：历史管理员凭据，或通过 `memberId` 关联统一账号的后台角色、可管理模块和负责部门。
- `inventory.json`、`inventory-ledger.json`：物资与出入库流水。
- `funds.json`：可归档资金账户与资金流水；删除账户时保留流水并核销剩余余额。
- `usage-requests.json`：材料和资金使用审批。
- `email-approval-tokens.json`：一次性邮件审批 Token 哈希及使用状态。
- `member-activation-codes.json`：成员一次性激活码 HMAC 哈希、过期和使用状态。
- `member-messages.json`：成员向负责管理员发起的问询线程和回复。
- `notifications.json`、`audit.json`：通知历史与审计记录。

## 数据库迁移时机

出现多实例、高并发写入、复杂统计查询、跨表事务或数据量显著增长时，优先迁移 PostgreSQL。迁移后应为项目分类、学号、申请状态和流水外键建立索引，并以数据库事务替换 `withResourceLock()`。
