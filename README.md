# 科创中心官网与运营系统

科创中心的公开官网、成员中心和运营后台。前端为无构建步骤的原生 HTML/CSS/JavaScript，后端为 Node.js + Express。

## 数据存储

当前没有使用关系型数据库，运行数据保存在 `DATA_DIR` 指定目录下的 JSON 文件中，邮件配置和资源密码使用 AES-256-GCM 加密 JSON 保存。生产环境使用 `/var/lib/tech-club`，文件权限为 `600`。

这种方案部署简单，适合当前中小规模社团；写入采用临时文件原子替换，库存和资金审批使用进程内串行锁。它不适合多实例部署或大量并发，达到该规模时应迁移到 PostgreSQL 或 SQLite，并使用数据库事务替代进程锁。

主要数据文件包括内容、申请、成员、管理员、通知、审计、库存、资金和审批记录。运行数据、上传文件、环境变量和账号密码不会进入 Git。

## 功能入口

- 统一入口：`/portal.html`
- 官网：`/`
- 后台工作区总览：`/admin.html`
- 运营宣发后台：`/admin.html?workspace=operations`
- 人员管理后台：`/admin.html?workspace=people`
- 资源与资金后台：`/admin.html?workspace=assets`
- 加入申请：`/join.html`
- 资源中心：`/resources.html`
- 成员中心：`/member.html`

项目支持独立分类、名称/编号/分类/标签/简介即时搜索，以及分类筛选。后台同样可以搜索和筛选项目，并更新每个项目的分类、标签、媒体、主题色和链接。每个资源节点可配置主链接和多个具名附加链接，也可作为最多三级的合集包含多个子资源；子资源继承祖先合集权限，并可拥有自己的权限标识和加密提取码。

加入申请要求填写姓名、学号、班级、联系方式、申请部门和申请理由，并明确确认招新信息使用说明。申请转为成员后，学号和班级会保留在成员档案中。

物资和资金使用必须由成员提交申请，经主管理员或审核员批准后才会在串行锁中校验并扣减。所有访问、申请、审批、库存和资金操作均写入审计记录。

物资可由编辑归档或重新启用，资金账户可由主管理员归档或重新启用；归档后不会再出现在成员的新申请选项中。主管理员可以删除物资或资金账户，删除会保留历史流水，剩余库存或余额会自动写入核销流水，存在对应待审批使用申请时禁止删除。

新成员账号由“部门账号简写 + 学号”自动生成，例如软件与算法部 `S` 加学号 `2024130110` 得到 `S2024130110`。系统签发 10 位随机一次性激活码，默认 7 天有效；成员在独立的 `/activate.html` 页面使用账号、激活码和自选密码完成激活，激活成功后返回登录页手动登录，系统不会自动建立成员会话。激活前不能登录、读取内部资源或被授予后台权限。激活码只返回或发送一次，服务器仅保存 HMAC 哈希，并支持管理员重新签发。管理员不再创建第二套登录凭据，而是在现有成员身份上附加后台角色、模块和部门范围；没有后台授权的成员登录管理端会返回 `403`。现有历史管理员和成员账号保持兼容。

只有 `owner` 自动拥有全部后台权限；其他管理员由主管理员显式分配“可管理模块”和“负责部门”，后台不会加载或显示未授权面板，申请、成员、通知和使用审批数据同时按部门过滤。成员中心由服务端根据资源权限、`material.request` 和 `fund.request` 生成可见模块，未授权资源、库存和资金账户不会返回浏览器。

加入申请通知按管理员身份选择具体负责人。普通系统通知仍可配置默认邮箱组，但申请通知不会再通过匿名邮箱字符串识别负责人；未配置时默认通知所有主管理员。

通知中心是所有启用管理员的基础能力。非主管理员只能向负责部门内的有效成员和尚未转为成员的申请人发送通知，可选择全部申请人或逐个勾选；主管理员还可以通知全体管理员、默认邮箱组和外部邮箱。所有收件地址按邮箱去重。

每张未转成员申请卡都提供“通知此申请人”入口，并自动在通知中心选中对应邮箱。管理员可以在申请转为成员前修正通过/拒绝结果，系统会重新发送结果通知并记录审计。

成员中心提供“联系管理员”问询线程。成员提交的问题会通知负责其部门的管理员；所有管理员可在通知中心查看自己负责范围内的问询、回复并结束线程。成员可在原线程继续补充或回复，已结束线程会随成员的新回复重新打开并再次通知管理员。管理员回复保存在成员中心，有成员邮箱时同步发送邮件。

每封加入申请通知邮件都包含绑定具体管理员、申请和动作的一次性“通过/拒绝”链接，使用与材料/资金审批相同的无登录确认页和失效规则。申请收件箱也提供明确的通过、拒绝按钮和审批意见；只有已通过申请才能转为成员。加入申请通过与拒绝的结果邮件模板可在邮件设置中分别编辑。

加入申请限流以“学号 + 联系方式”作为申请人标识，每位申请人 30 分钟最多成功提交 3 次；共享出口 IP 仅承担每 10 分钟 30 次的宽松防滥用限制。表单校验失败、服务器写入失败和 `429` 重试不会占用或延长额度。

材料和资金申请会向当前有审批权限且负责该部门的管理员分别发送一次性审批邮件。批准和拒绝链接绑定管理员、账号版本、申请和动作，24 小时过期；浏览器先打开无登录确认页，只有 POST 确认才进入串行审批锁。Token 通过 URL Fragment 传递，服务器仅保存 HMAC 哈希，不进入 Nginx URL 日志。申请已处理、账号停用、邮箱或权限范围变化后链接立即失效，邮件审批审计会记录审批人、时间、来源和 Token 记录 ID。

审批完成后会向成员当前邮箱发送结果邮件。批准与拒绝分别拥有可编辑的主题和正文模板，可使用申请人、编号、类型、对象、数量金额、用途、结果、审批意见、审批人和审批时间变量；后台审批和邮件链接审批共用这些模板。

## 本地运行

安装并启动 API：

```bash
cd backend
npm ci
ADMIN_PASSWORD='replace-with-a-local-password' SESSION_SECRET='replace-with-a-random-secret' PUBLIC_BASE_URL='http://localhost:8080' npm start
```

启动静态页面：

```bash
python3 -m http.server 8080 --directory public
```

运行检查与端到端测试：

```bash
cd backend
npm run check
npm run test:applications
npm run test:admin-scope
npm run test:email-approval
npm run test:permissions
npm run test:resources
```

## Git 工作流

功能开发从主分支创建短分支，提交前运行后端检查和端到端测试。运行数据、上传内容、`.env`、SMTP 授权码及 `docs/TEST_ACCOUNTS.md` 均由 `.gitignore` 排除，禁止提交到仓库。

部署配置位于 `deploy/`。生产静态目录只接收 `public/` 的内容，后端只部署 `backend/`，避免源码、测试账号和运维文件暴露在 Web 根目录。
